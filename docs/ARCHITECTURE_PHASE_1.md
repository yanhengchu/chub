# Hub 第一阶段技术架构

## 1. 文档目的

本文档将第一阶段 PRD 转换为可实施的技术方案，明确系统边界、模块职责、关键数据流、安全约束和验证方式。

本文档只覆盖第一阶段，不为多节点聚合、任务转发或统一控制台预先设计复杂架构。

## 2. 架构目标

- 使用同一套工程和代码运行于 macOS 与 Ubuntu。
- 每个 Hub Node 可以独立启动、访问和执行本机能力。
- 浏览器页面与未来客户端复用同一组受控服务接口。
- 系统命令只能由已注册任务以固定方式调用。
- 平台差异集中在平台检测和平台任务中，不扩散到 API 与页面。
- 保持模块简单、可测试，并支持后续增加节点能力。

## 3. 系统边界

```text
浏览器 / 后续客户端
        │
        ▼
  Hub Node Web 服务
        │
        ├── 健康检查
        ├── 节点状态
        ├── 任务列表与执行
        ├── Hub 日志
        └── Web 页面与静态资源
                │
                ▼
      本机系统与白名单程序
```

第一阶段每台设备运行一个 Hub Node。节点之间没有通信关系，也不存在中心服务。

## 4. 技术基线

- 运行时：Python 3.12 及以上。
- Web 框架：FastAPI。
- 服务运行：Uvicorn。
- 配置格式：YAML，敏感值由环境变量提供。
- 数据模型与校验：Pydantic。
- 页面：Jinja2、HTML、CSS 和少量原生 JavaScript。
- 系统信息：psutil。
- 测试：pytest 与 FastAPI 测试客户端。

第一阶段不引入数据库、缓存、消息队列、前端构建工具或容器 SDK。

## 5. 工程结构

```text
chub/
├── main.py
├── pyproject.toml
├── requirements.txt
├── README.md
├── app/
│   ├── api/           # 受控服务接口
│   ├── core/          # 配置、日志、平台、安全、响应
│   ├── services/      # 系统状态与日志读取
│   ├── tasks/         # 任务模型、注册、执行与平台任务
│   └── web/           # 页面、模板与静态资源
├── config/            # 非敏感节点配置
├── docs/
├── logs/              # 运行时生成，不提交日志文件
└── tests/
```

模块内部按职责拆分，避免将系统采集或命令执行直接写在接口路由中。

## 6. 应用生命周期

Hub Node 启动时按以下顺序初始化：

1. 定位项目根目录和配置文件。
2. 读取 YAML 配置，并从环境变量注入敏感配置。
3. 使用 Pydantic 校验完整配置。
4. 检测当前操作系统，与节点声明类型进行比对。
5. 初始化控制台和文件日志。
6. 注册公共任务和当前平台任务。
7. 创建应用并挂载服务接口、页面和静态资源。
8. 开始监听节点配置指定的地址和端口。

配置格式无效等不可恢复错误应阻止启动并给出明确原因；平台声明不一致先记录警告，平台任务以实际检测结果为准。未设置 Token 时记录醒目警告并允许基础服务启动，公开健康检查保持可用，受保护接口统一返回服务未配置错误，不得降级为免认证访问。

## 7. 配置设计

配置分为两类：

- 非敏感配置：应用版本、节点身份、节点类型、监听地址、日志和任务超时等，保存在 YAML 中。
- 敏感配置：访问凭证，通过环境变量提供，不写入仓库。

每台设备使用自己的部署配置，但共享同一套代码。仓库提供不含真实凭证的示例配置。

配置路径默认相对项目根目录解析，同时允许通过环境变量指定配置文件，便于不同节点部署和测试。

## 8. 平台适配

平台层输出统一的平台标识：

- `macos`
- `ubuntu`
- `windows`
- `unknown`

第一阶段只承诺 macOS 和 Ubuntu。

公共状态和任务不依赖具体平台模块。平台专属任务通过注册表按实际平台装载：macOS 注册 Codex 检查，Ubuntu 注册 Docker 检查。未支持平台可以启动基础服务，但不加载平台专属任务。

## 9. 服务接口

第一阶段提供以下能力接口：

- 服务健康检查。
- 节点状态读取。
- 当前节点可用任务列表。
- 白名单任务执行。
- Hub 近期日志读取。
- 节点首页。

具体 HTTP 路径沿用第一阶段设计方案：

```text
GET  /
GET  /api/health
GET  /api/status
GET  /api/tasks
POST /api/tasks/run
GET  /api/logs
```

接口只处理认证、输入校验、业务调用和响应转换。所有 JSON 接口使用一致的成功或失败结构；具体字段在实现模型中定义并由测试固定。

健康检查不需要认证且不返回敏感节点信息。其他 API 需要访问凭证。

## 10. 节点状态

系统状态服务负责采集并标准化：

- 节点 ID、节点名称和节点类型。
- 主机名、操作系统和版本。
- CPU 使用率。
- 内存容量和使用率。
- 磁盘容量和使用率。
- 系统启动时间和运行时长。
- Hub 版本和当前时间。

API 和任务都复用该服务，避免产生两套状态采集逻辑。采集失败应返回受控错误，不泄露调用栈或本机敏感路径。

## 11. 任务架构

### 11.1 任务定义

内部任务定义至少包含：

- 唯一名称。
- 展示标题和说明。
- 支持的平台范围。
- 最长执行时间。
- 固定的处理函数。
- 受校验的参数模型。

内部定义中的处理函数、命令和实现细节不得通过任务列表接口返回。
任务列表只公开名称、标题、说明、支持平台、超时时间和必要的参数
Schema。第一阶段四个任务均不接受业务参数，使用拒绝额外字段的空参数模型。

### 11.2 任务注册

任务注册表负责：

- 注册公共和平台任务。
- 拒绝重复任务名。
- 按名称查询任务。
- 列出当前平台可用任务。
- 拒绝未注册任务和平台不匹配任务。

注册表由应用启动过程创建并保存在当前 FastAPI 应用实例中，不使用模块级
可变全局注册表。所有白名单任务定义都进入注册表，任务列表按实际平台过滤，
执行器再次校验平台；这样可以区分未知任务的 `404` 与已知但平台不支持任务
的 `409`。`unknown` 和第一阶段未承诺的 Windows 只能列出和执行公共任务。
重复任务名属于启动配置错误，应立即阻止应用启动。

### 11.3 任务执行

任务执行器负责统一完成：

1. 查询任务。
2. 校验平台与参数。
3. 在最长执行时间内调用处理函数。
4. 捕获超时和异常。
5. 记录不含敏感参数的执行日志。
6. 返回统一任务结果。

任务执行结果状态固定为：

- `success`：任务完成并得到有效检查结果。
- `failed`：任务完成，但目标程序不存在、服务不可用、权限不足或检查失败。
- `timeout`：外部检查超过任务规定时间。

任务已被合法接受后，以上三种状态均通过成功的 API 响应返回。任务不存在、
平台不支持和参数非法属于请求错误，分别返回稳定的 API 错误。单个任务失败
或超时不得影响健康检查、状态接口和其他任务。

第一阶段不使用线程超时模拟强制取消。通用 Python 任务必须是短小、同步且
可信的内置处理函数；可能阻塞的外部检查通过受控命令运行器执行，由子进程
超时机制负责终止。第一阶段不引入任务队列、历史、定时执行、后台任务、
多进程任务沙箱或任务依赖编排。

### 11.4 受控命令运行

涉及外部程序时统一使用受控命令运行器：

- 命令和参数必须由任务代码以固定字符串列表提供。
- 使用 `shell=False`，禁止 Shell 解释。
- API 参数不得指定或拼接命令、参数、环境变量、工作目录和可执行文件。
- 使用子进程超时并在超时后确保进程退出。
- 捕获并限制标准输出和错误输出，单次最多保留 64 KiB。
- 统一处理程序不存在、权限不足、非零退出码和超时。
- 日志只记录任务名、结果状态和耗时，不记录完整参数和命令输出。

第一阶段任务只查询状态，不执行启动、停止、重启、安装、拉取或修改配置等
操作。

### 11.5 任务接口

任务执行请求固定为：

```json
{
  "task": "check_system",
  "params": {}
}
```

请求模型拒绝额外字段。合法任务无论检查成功、失败或超时，均返回：

```json
{
  "success": true,
  "data": {
    "task": "check_system",
    "status": "success",
    "message": "System status collected",
    "duration_ms": 12,
    "result": {}
  }
}
```

`result` 由具体任务定义，失败或超时时可以为 `null`。第一阶段不返回任务
执行 ID，也不保存最近结果或历史记录。

## 12. 首批任务

- `show_version`：返回 Hub 版本、Python 版本、节点 ID 和实际平台。
- `check_system`：复用 M2 状态服务，返回 CPU、内存、磁盘和运行时长。
- `check_codex`：仅 macOS，查找固定程序名 `codex`，使用固定
  `--version` 参数，返回是否可用、程序路径和版本。
- `check_docker`：仅 Ubuntu，使用固定 Docker CLI 参数，返回 CLI、客户端、
  服务端和 Compose 可用性及容器数量概况，不返回容器环境变量、挂载详情、
  日志或完整 `docker info`，也不执行管理操作。

外部程序不存在、权限不足或服务未运行时，应作为可理解的任务结果返回，不导致 Hub Node 异常退出。

## 13. 认证与安全

- 除健康检查外，第一阶段所有 API 均为受保护接口，统一使用 Bearer Token。
- 不提供关闭认证的配置开关；本地开发也通过测试 Token 访问受保护接口。
- 服务端只从 `HUB_TOKEN` 环境变量读取真实 Token，并使用 Pydantic `SecretStr` 保存，避免配置对象在日志或调试输出中直接展示凭证。
- 空字符串或全空白 Token 视为未配置。未设置 Token 时允许节点启动并记录警告，健康检查保持可用，受保护接口统一返回 `503 security_not_configured`。
- 缺失 Authorization 时返回 `401 authentication_required`；格式错误或 Token 不匹配时返回 `401 invalid_credentials`。所有 `401` 响应携带 `WWW-Authenticate: Bearer`。
- Token 比较使用 `secrets.compare_digest`，认证实现不得记录原始 Header、Token 或完整配置对象。
- 建议使用至少 32 字节随机数据生成 Token。Token 强度不足时记录警告但不阻止第一阶段启动，避免将格式要求误当作真正的熵校验。
- Token 通过修改环境变量并重启节点完成轮换。第一阶段不实现刷新令牌、多 Token、用户和权限系统。
- 客户端可选择将 Token 保存在当前浏览器本地；页面输入使用密码类型，不明文回显已保存值。
- 请求日志不得记录 Authorization 请求头。
- 错误响应不得包含 Token、调用栈或无必要的本机路径。
- 任务参数由明确的数据模型校验，拒绝额外的命令类字段。
- 日志接口只接受行数，不接受文件路径；行数具有固定上下限。

本方案仍以可信网络部署为前提，不替代 TLS、互联网边界防护或完整用户体系。

成功响应保持：

```json
{
  "success": true,
  "data": {}
}
```

错误响应统一为：

```json
{
  "success": false,
  "error": {
    "code": "invalid_credentials",
    "message": "Invalid access token"
  }
}
```

错误响应不同时返回 `data`。`code` 是供客户端判断的稳定标识，
`message` 是不包含调用栈、凭证和本机敏感路径的简短说明。

## 14. 日志设计

Hub Node 同时输出控制台日志和滚动文件日志，记录：

- 应用启动和节点信息。
- 受控的请求概要。
- 任务名称、结果和耗时。
- 配置、状态采集和任务异常。

日志读取服务只读取配置中已确定的 Hub 日志文件，并限制返回行数。日志文件和轮转产物不提交到 Git。

## 15. Web 页面

服务端返回页面骨架，浏览器通过受控 API 获取状态、任务和日志。

页面负责：

- 录入、保存、更换和清除当前节点 Token。
- 展示节点身份和系统概况。
- 展示当前平台任务并触发执行。
- 展示执行中、成功、失败和超时状态。
- 查看近期日志。

页面不拼装系统命令，也不直接接触本机系统。第一阶段优先保证手机端可读和操作清晰。

## 16. 错误处理

错误分为：

- 认证失败。
- 输入校验失败。
- 任务不存在或平台不支持。
- 任务执行失败或超时。
- 状态采集或日志读取失败。
- 服务内部异常。

每类错误使用稳定的状态码和机器可识别错误标识，并返回面向用户的简短说明。内部异常写入日志，外部响应不暴露实现细节。

M3 请求错误固定为：

- 未注册任务：`404 task_not_found`。
- 当前平台不支持：`409 task_not_supported`。
- 整个 API 请求体缺字段或包含额外字段：`422 invalid_request`。
- 已识别任务的 `params` 非法：`422 invalid_task_parameters`。

目标程序不存在、服务未运行、检查命令失败和超时不是 API 内部错误，而是
HTTP 200 中的任务 `failed` 或 `timeout` 结果。

## 17. 测试策略

### 17.1 单元测试

- 配置读取、环境变量覆盖和缺失字段。
- macOS、Ubuntu 与未知平台检测。
- 状态数据结构。
- 任务注册、平台过滤和重复注册。
- 任务成功、失败、超时和不支持平台。
- 任务列表不暴露处理函数、命令和内部实现信息。
- 四个首批任务拒绝额外参数。
- 受控命令运行器不启用 Shell，限制输出并真实终止超时子进程。
- 两个独立任务并发执行时不共享可变结果。
- 单个任务失败或超时后，健康检查和状态接口仍然可用。
- 日志尾部读取和行数边界。

### 17.2 API 测试

- 健康检查无需 Token。
- 有效 Token 可以访问受保护接口。
- 缺失、格式错误和不匹配的 Token 返回稳定的 `401` 错误结构。
- 服务端未配置 Token 时，受保护接口返回 `503`，健康检查仍可公开访问。
- Token 不出现在响应、控制台日志和文件日志中。
- 状态、任务列表、任务执行与日志响应结构。
- 未注册任务、非法参数和超时结果。
- 日志接口不能选择任意路径。

### 17.3 页面与手工测试

- 手机尺寸下的主要信息和操作。
- Token 录入、保存、更换、清除和错误提示。
- macOS 上的 Codex 检查。
- Ubuntu 上的 Docker 检查。
- 通过可信网络分别访问两个节点。
- 一台设备离线时另一节点保持可用。

## 18. 部署与运行

每台设备：

1. 获取同一版本代码。
2. 准备 Python 3.12 及以上环境和依赖。
3. 创建该节点的非敏感配置。
4. 通过环境变量设置访问 Token。
5. 启动 Hub Node，并根据需要配置操作系统级自启动。

第一阶段先支持直接运行 Python 服务。系统服务、自启动和升级流程可以在节点功能稳定后补充，但部署文档必须说明手工启动和验证方式。

## 19. 第一开发里程碑

首个可运行里程碑包含：

- Python 工程与依赖声明。
- 基础目录结构。
- YAML 配置读取和校验。
- 环境变量 Token 注入。
- 平台检测。
- 控制台与文件日志初始化。
- FastAPI 应用启动。
- 无认证健康检查接口。
- 配置和健康检查自动化测试。

该里程碑不包含状态、任务、日志 API 和 Web 管理页面，它们在后续任务中递增实现。
