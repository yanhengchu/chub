# 配置驱动的网页下载自动化方案

> 状态：首版基础能力已实现，真实网站配置和下载验收待完成。

## 1. Chub 首页交互卡片

首页在 `Codex PTY` 之后、`设计文档` 之前增加“自动化任务”卡片，作为任务状态和常用操作入口，不在首页展开配置及执行日志。

```text
自动化                                      刷新
自动化任务
按计划复用浏览器登录状态执行固定任务

Debug Chrome：已连接       已启用 2 个任务

月度报表下载          上次成功 · 今天 08:30      运行
资料归档下载          下次执行 · 明天 09:00      运行

                                        全部任务
```

- 标题右侧只保留本卡片“刷新”，并且只更新自动化数据；“全部任务”放在卡片底部右侧进入次级页面。
- 桌面端双列布局使用卡片自身内容高度，内容从顶部连续排列，不随同排较高卡片拉伸；手机端单列同样保持自然高度。
- 卡片独立加载和展示错误；任务运行、状态轮询及浏览器启停只更新自动化卡片，不触发其他卡片刷新。
- 卡片展示 Debug Chrome 状态、启用任务数量，以及最近或即将执行的少量任务；尚未接入调度时只展示最近结果。
- 每个任务行展示名称、最近结果或下次执行时间，并提供一个“运行”按钮；浏览器未运行或任务已经执行时禁用该按钮。
- 点击“全部任务”进入次级页面；手动运行、运行中、成功和失败都应有明确反馈。
- 完整任务列表和最近结果放在次级页面，配置仍由本机 YAML 管理，首页保持简洁。

卡片内单独提供“浏览器环境”行，它是所有任务的运行前提，不属于 YAML 任务：

- 未启动时展示“启动”，默认启动有界面模式，便于人工登录和处理验证码。
- 已运行时展示模式和“停止”；自动化任务执行期间禁止停止。
- 启停按钮等待 Debug Chrome 最终状态后再提示成功，失败时展示 profile 未初始化、端口冲突等明确原因。
- 启动和停止都写入操作日志；不会影响用户日常 Chrome，只管理 `chrome-cdp` 拥有的独立实例。

## 2. 核心主流程

1. 系统定时、首页手动操作或命令行触发指定任务。
2. 连接已启动的 Debug Chrome，复用现有登录状态并创建任务专用页面。
3. 按任务配置依次完成页面操作并触发下载。
4. 捕获下载文件，完成内容校验和安全落盘。
5. 记录成功或失败结果，并在首页及“全部任务”页面更新状态。
6. 清理任务页面和连接，不关闭共享 Debug Chrome，也不影响用户已有页面。

所有入口必须调用同一个 Runner：

```text
首页 / 命令行 / 系统定时器
            ↓
        Automation Runner
            ↓
     状态文件 / 下载文件 / 日志
```

首页手动运行只负责创建后台 Runner 并返回“已受理”，随后轮询状态文件展示最终结果，不能把后台进程成功启动当作任务成功。每个任务使用原子更新的 JSON 状态文件保存排队、运行、成功和失败状态，服务重启后仍可读取；首版不引入数据库。

## 3. 目标与边界

通过配置描述固定网站的页面操作流程，复用已启动的 Debug Chrome 登录状态，完成文件下载、校验和安全落盘。

首版聚焦单网站、单任务、单文件下载。网站差异放在配置中，浏览器连接、下载处理、日志和任务互斥由公共执行器统一负责。

## 4. 配置示例

网站和下载流程统一配置在本机文件 `config/automations.local.yaml`：

```yaml
version: 1

tasks:
  monthly-report:
    name: 月度报表下载
    enabled: true

    browser:
      session: debug-chrome
      start_url: https://example.com/reports
      allowed_hosts:
        - example.com

    login:
      check:
        selector: "[data-testid='user-avatar']"
        timeout_ms: 10000
      expired_message: 登录状态已失效，请重新登录 Debug Chrome

    steps:
      - action: wait
        selector: "#report-list"
        timeout_ms: 15000

      - action: hover
        selector: "[data-report='monthly']"

      - action: click
        selector: "button.open-download-menu"

      - action: click
        selector: "button.download-pdf"
        expect: download
        timeout_ms: 60000

    output:
      directory: monthly-report/downloads
      filename: "monthly-report-{date:%Y-%m}.pdf"
      conflict: replace
      timezone: Asia/Shanghai

    validation:
      non_empty: true
      extensions:
        - .pdf
      min_bytes: 1024
      signature: pdf

    execution:
      timeout_ms: 120000
      lock_timeout_ms: 1000
      safe_step_retries: 1
```

## 5. 配置约定

### `browser`

- `session`：首版固定使用 `debug-chrome`。
- `start_url`：任务入口页面。
- `allowed_hosts`：允许主页面导航到的域名白名单。

域名白名单只检查 `start_url`、配置中的 `goto`、主页面跳转和任务新开的弹窗目标。图片、脚本、字体和 CDN 等子资源不按该白名单拦截，避免破坏网站正常加载。发现主页面或弹窗导航到白名单外时终止任务。

### `login`

- `check.selector`：进入任务页面后用于确认登录状态的元素。
- `check.timeout_ms`：登录状态检查超时。
- `expired_message`：登录失效时返回的明确提示。

登录密码、Token 和 Cookie 不写入 YAML，登录状态由独立 Debug Chrome profile 保存。

### `steps`

步骤严格按声明顺序执行。首版支持：

- `goto`：导航到指定地址。
- `wait`：等待元素进入指定状态。
- `hover`：移动鼠标到指定元素。
- `click`：点击元素。
- `dispatch_event`：向元素派发固定名称的浏览器事件。

触发下载的步骤必须显式设置 `expect: download`，执行器不再假定最后一步必然触发下载。首版每个任务只允许一个下载步骤。

配置不允许执行任意 JavaScript、Shell 或 Python。等待优先使用元素状态和页面条件，不使用固定时长休眠作为常规同步手段。

配置模型必须拒绝未知字段、未知动作和不适用于该动作的参数，避免拼写错误被静默忽略。

### `output`

- `directory`：相对于自动化数据根目录的任务下载目录，解析后的路径必须仍位于该根目录内。
- `filename`：最终文件名，允许使用受控的日期变量。
- `conflict`：文件冲突策略；首版支持 `replace`、`skip` 和 `fail`。
- `timezone`：日期变量使用的 IANA 时区，必须显式配置；首版不依赖机器本地时区隐式生成文件名。

浏览器提供的文件名和配置生成的文件名都必须经过路径清理，禁止绝对路径和目录穿越。

### `validation`

- `non_empty`：拒绝空文件。
- `extensions`：允许的文件扩展名。
- `min_bytes`：最小文件大小。
- `signature`：基础文件签名校验；例如 `pdf` 必须以 `%PDF-` 开始。

扩展名和大小不能证明文件内容有效。例如登录失效时，网站可能返回 HTML 错误页并使用 `.pdf` 文件名。首版对已声明的文件类型执行基础签名校验；不支持的类型必须明确失败或在任务实现前补充校验器，不能静默退化为只看扩展名。

### `execution`

- `timeout_ms`：整个任务的最大执行时间。
- `lock_timeout_ms`：等待共享浏览器锁的最大时间；超时返回 `busy`，不进入浏览器流程。
- `safe_step_retries`：只用于导航、等待元素等明确无副作用步骤的有限重试次数。

首版不支持整个任务自动重试，也不重试 `click`、`dispatch_event` 和下载触发步骤。下载按钮点击后，即使没有捕获到下载事件，也可能已经产生服务端副作用或浏览器下载；此时任务应明确失败并等待人工判断，不能从头再次执行。

共享 Debug Chrome 自动化锁属于执行器的强制安全能力，不由单个网站配置关闭。锁使用跨进程的操作系统文件锁，进程退出后由操作系统自动释放；状态文件只用于展示，不作为锁本身。

## 6. 文件落盘规则

文件处理由公共执行器负责，不作为页面步骤暴露：

```text
捕获下载事件
  → 保存到目标目录内的临时文件
  → 校验非空、大小和类型
  → 应用冲突策略
  → 跨平台原子替换为最终文件
  → 记录执行结果
```

- 临时文件与目标文件位于同一目录，确保最终替换不跨文件系统。
- `replace` 使用 `os.replace` 等跨平台原子替换语义；`skip` 和 `fail` 在落盘前检查目标文件，不能用普通 `rename` 模拟覆盖。
- 校验失败或执行异常时清理本次任务的临时文件。
- 下载目录使用仅当前用户可访问的权限，临时文件和最终文件统一设置为 `600`；Windows 上使用平台可提供的等效保护。
- 最终文件写入成功后再把任务标记为成功。
- 可在后续通过文件哈希和状态文件实现内容去重。

## 7. 配置与调度边界

`config/automations.local.yaml` 只描述任务、网站、页面操作和下载结果，不承载平台定时表达式；示例配置使用 `config/automations.example.yaml`，本机真实配置不提交。

调度配置独立管理，负责定义任务 ID、执行时间和启用状态，再调用统一任务入口。macOS、Ubuntu 和 Windows 的系统调度器只负责触发任务，不直接包含网站操作逻辑。

任务要求 Debug Chrome 已经由用户或独立生命周期机制启动。任务只检查状态，未启动时返回明确失败，不在后台偷偷启动、停止、切换 profile 或改变 headed/headless 模式。

首版先提供命令行和首页手动触发。系统定时器在真实网站流程稳定后接入；接入前需要明确 Debug Chrome 是用户登录后常驻，还是允许定时任务在浏览器未运行时明确失败。

## 8. 日志与执行结果

每次执行至少记录：

- 任务 ID、开始时间、结束时间和最终状态。
- 当前步骤和失败原因。
- 下载文件名、大小和最终路径。
- 是否发生重试、跳过或覆盖。

日志不得记录 Cookie、Authorization、页面敏感字段或完整浏览器上下文。配置校验错误、登录失效、下载超时、文件校验失败应使用可区分的错误状态。

Web 触发的 Runner 完成后必须主动写入最终 `succeeded` 或 `failed` 操作日志，不能依赖页面继续轮询；Manager 读取状态时仅负责补偿异常退出或旧状态遗漏的最终日志。Runner 子进程不继承 Hub Token 的值。

## 9. 首版范围

包含：

- 复用一个已启动的 Debug Chrome。
- 每次执行跟踪并关闭自己创建的页面和弹窗，不操作用户已有页面。
- YAML 配置驱动的串行页面步骤。
- Playwright 标准 `Download` 事件。
- 单任务单文件下载。
- 登录状态检查、共享浏览器跨进程互斥、任务超时和安全步骤有限重试。
- 临时文件、基础文件签名校验和原子落盘。
- 命令行手动执行，为后续系统定时任务提供统一入口。

暂不包含：

- 验证码和多因素认证自动处理。
- 任意脚本执行。
- 多下载步骤或并行下载。
- PDF 预览器、Blob、接口直链等非标准下载适配。
- 自动启动、停止或接管用户日常 Chrome。
- 跨设备任务编排和集中式任务队列。

## 10. 验收标准

- 配置合法时，可以复用 Debug Chrome 的既有登录状态完成一次真实下载。
- 页面步骤严格按配置执行，下载步骤无需依赖“最后一步”约定。
- 登录失效、元素超时和未触发下载时有明确结果，不产生最终文件。
- 下载内容先写入临时文件，校验通过后才原子生成目标文件。
- 空文件、错误类型、非法路径和越界域名会被拒绝。
- 同一任务重复触发时不会并发执行。
- 不同任务复用同一 Debug Chrome 时也不会并发操作浏览器。
- 任务只操作并清理自己创建的页面和弹窗，用户已有页面在成功和失败路径中都不受影响。
- Playwright 断开后 Debug Chrome 及其中的登录状态仍然保留。
- 下载触发步骤失败时不会自动从头重试，避免重复产生下载或服务端副作用。
- `.pdf` 等已声明类型会进行基础文件签名校验，HTML 错误页不会被当作成功文件保存。
- 日志能够定位失败步骤，且不泄露登录凭据和浏览器敏感状态。
