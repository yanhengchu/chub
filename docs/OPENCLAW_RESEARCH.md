# OpenClaw 方案调研

> 状态：初步调研，重点分析 OpenClaw 的主体架构，以及它与 Chub 后续发展的关系。

## 1. 简介

[OpenClaw](https://openclaw.ai/) 是一个开源、自托管的个人 AI 助手平台。它通过常驻 Gateway 连接消息渠道、AI Agent、模型、工具和设备节点，使用户能够从 Web 或聊天应用发起请求，再由 Agent 在授权范围内完成任务。

OpenClaw 的核心不是单一聊天页面，而是一个以 Agent 为中心的运行与连接平台。

## 2. 主体架构

```text
消息渠道 / Web / 移动端
            │
            ▼
     OpenClaw Gateway
            │
     ┌──────┴──────┐
     ▼             ▼
 Agent 与模型    会话与路由
     │
     ▼
工具 / 技能 / 自动化
     │
     ▼
本机能力 / 外部服务 / 设备节点
```

### 2.1 Gateway

Gateway 是 OpenClaw 的中心服务，也是整个系统的状态与连接中枢，主要负责：

- 接收 Web 和消息渠道中的请求。
- 管理频道账号及设备连接。
- 将请求路由到对应 Agent 和会话。
- 协调模型调用、工具执行与节点调用。
- 维护会话、配对、任务和运行状态。
- 提供 Control UI、CLI、诊断和运维入口。

OpenClaw 采用中心化 Gateway 结构。通常由一台长期在线的电脑或服务器运行 Gateway，其他入口和设备连接到该 Gateway。

### 2.2 Agent 与模型

Agent 是请求的理解和执行主体。它结合模型、系统说明、工作区资料、会话上下文和可用工具，判断如何回应或完成任务。

一个 Gateway 可以运行多个 Agent。不同 Agent 可以拥有独立的角色、工作区、会话、模型、记忆和工具权限。多 Agent 的主要价值是隔离场景和权限，例如把个人事务、开发工作和家庭设备分别交给不同 Agent，而不是让一个助手拥有全部数据和能力。

OpenClaw 本身不是大语言模型。任务质量、成本和隐私仍取决于实际使用的云端或本地模型。

### 2.3 Channel

Channel 是用户访问 OpenClaw 的交互入口，包括 WebChat，以及 Telegram、WhatsApp、Discord、Slack、Signal、iMessage 等消息渠道。

Channel 层负责接收消息、识别发送者和群组来源、将消息路由到 Agent 或会话，并把结果返回原渠道。访问范围可通过白名单、配对和群组规则限制。

Channel 只应承担消息接入和身份边界，不应直接获得设备的全部执行权限。

### 2.4 会话、上下文与记忆

Gateway 统一管理会话。不同联系人、群组或 Agent 可以使用隔离会话，避免上下文相互污染。

OpenClaw 中的信息可以分为三层：

- 当前会话：当前任务和近期对话。
- 工作区资料：Agent 的固定说明、项目背景和可维护知识。
- 长期记忆：经过筛选、需要跨会话保留的用户偏好和事实。

长期记忆不应等同于保存全部聊天记录。用户需要能够查看、修正和删除重要记忆，敏感凭证则不应进入普通记忆或提示上下文。

### 2.5 工具、技能与插件

工具是 Agent 可以调用的基础动作，例如访问文件、查询网页、调用服务或执行系统能力。技能在工具之上描述一类可重复任务，通常包含使用说明、脚本和相关资源。插件则用于扩展频道、工具、模型提供方或其他运行能力。

```text
工具：一个具体动作
技能：完成一类任务的方法
插件：向系统增加新的能力类型
```

工具和技能让 OpenClaw 从对话助手扩展为执行型 Agent，但也是主要风险来源。社区技能不能默认信任，安装前需要检查代码、依赖、凭证需求和系统权限。

### 2.6 自动化

OpenClaw 支持定时任务、Webhook 和其他事件触发方式，使 Agent 能够在没有即时聊天请求时主动运行，适合每日摘要、提醒、只读状态检查和固定资料整理。

自动化任务必须区分“已经触发”和“最终完成”。对于发送消息、修改数据、消费付费资源或控制设备的任务，需要限制重试，并根据风险增加人工确认。

### 2.7 Node

Node 是与 Gateway 配对的外部设备，可以是 macOS、iOS、Android 或无界面主机。节点向 Gateway 提供经过声明和批准的设备能力，例如相机、通知、Canvas、设备动作或受控系统命令。

```text
Gateway：接收消息、运行 Agent、路由工具调用
Node：在对应设备上执行已授权能力
```

Node 是 Gateway 的外设，不承担 Agent 和消息路由职责。节点需要经过设备配对，执行能力应受到命令范围、审批和本机权限控制。

### 2.8 管理与运维

OpenClaw 提供 Control UI、CLI、健康检查、日志、诊断和升级能力。由于它是自托管常驻服务，使用者仍需负责：

- Gateway 服务常驻和故障恢复。
- 模型、频道及设备凭证保护。
- 节点配对和权限回收。
- 工作区、记忆和配置备份。
- 版本升级及 Gateway 与节点的兼容性。
- 模型使用量和外部服务费用。

## 3. 核心功能

OpenClaw 的主体功能可以归纳为：

1. 通过 Web 和消息平台访问个人 AI 助手。
2. 使用多个模型和多个 Agent 处理不同场景。
3. 管理连续会话、工作区资料和长期记忆。
4. 通过工具、技能和插件执行实际任务。
5. 通过定时任务和事件触发主动运行。
6. 连接其他设备节点并调用设备能力。
7. 通过 Gateway 统一管理路由、配对、状态和运维。

它适合承载“从手机发起自然语言请求，由长期在线的 Agent 调用个人设备和服务完成任务”的使用方式。

## 4. 安全边界

OpenClaw 会同时接触外部消息、模型、个人数据和设备工具，安全重点不是单一登录认证，而是控制 Agent 能够调用什么。

主要风险包括：

- 外部消息、网页或文档通过提示注入诱导 Agent 调用工具。
- 高权限工具造成误删文件、错误发送消息或非预期系统操作。
- API Key、频道凭证和设备令牌泄露。
- 第三方技能、插件或依赖包含恶意行为。
- 长期记忆和日志保存过多敏感内容。
- Gateway 暴露到不可信网络。
- 定时任务在无人观察时重复产生副作用或费用。

适合个人环境的基本原则是：

- Gateway 默认只在本机或可信网络开放。
- 按 Agent 和场景提供最小工具权限。
- 读操作与写操作分离。
- 高风险、付费和不可逆操作保留人工确认。
- 凭证与工作区、记忆、页面和普通日志分离。
- 社区技能安装前进行检查。
- 节点离线或任务失败时不得自动扩大权限重试。

## 5. OpenClaw 与 Chub 的共同点

OpenClaw 和 Chub 都服务于个人设备的远程使用，存在以下共同方向：

- 都可以自托管在用户自己的电脑或服务器上。
- 都需要长期运行的后台服务。
- 都支持从手机访问电脑上的能力。
- 都有设备或节点的概念。
- 都需要认证、健康检查、日志和故障反馈。
- 都可以承载自动化、浏览器或系统操作。
- 都适合通过 Tailscale 等可信网络限制访问。
- 都强调用户掌控运行环境和数据。

两者在“让个人设备变成可远程使用的能力节点”这一目标上存在明显交集。

## 6. OpenClaw 与 Chub 的差异

| 维度 | OpenClaw | Chub |
| --- | --- | --- |
| 核心定位 | 个人 AI 助手与 Agent Gateway | 个人设备管理与受控操作服务 |
| 主要入口 | 消息渠道、WebChat、Control UI | 手机优先的 Web 管理页面 |
| 执行驱动 | Agent 理解请求并选择工具 | 用户选择固定功能，后端执行预设流程 |
| 设备结构 | 一个 Gateway 连接多个 Node | 当前每台设备独立运行一个 Chub |
| AI 能力 | 内建 Agent、模型、会话和记忆 | 不内建模型，Codex 由原生 CLI 管理 |
| 扩展方式 | 工具、技能、插件和节点能力 | 固定 API、白名单任务和配置模板 |
| 自动化特点 | 通用、灵活，可由 Agent 编排 | 面向真实场景逐项实现和验收 |
| 安全重点 | Agent 工具权限、配对、审批和沙箱 | Hub Token、Tailscale、固定命令与路径 |
| 结果特点 | 灵活，但模型参与带来不确定性 | 行为固定，最终状态更容易验证 |

最关键的差异是：OpenClaw 以 AI Agent 为中心，Chub 以受控设备操作为中心。

OpenClaw 的典型流程：

```text
自然语言请求
→ Agent 理解和规划
→ 选择工具或节点
→ 执行并回复
```

Chub 的典型流程：

```text
用户选择固定操作
→ 后端校验
→ 执行预设流程
→ 确认最终状态
```

## 7. 两者后续发展关系

OpenClaw 和 Chub 的功能存在交叉，但当前不适合合并成一个系统，也不建议由其中一个完全替代另一个。集成前应先确定每类能力的唯一责任方，避免同一个节点、任务或操作同时由两套系统管理。

### 7.1 职责划分原则

两者按“意图层”和“执行层”划分：

- OpenClaw 负责理解用户想做什么、组织上下文、选择可用工具并呈现结果。
- Chub 负责判断一个设备操作是否允许、如何安全执行，以及最终是否真正完成。

这意味着 OpenClaw 可以决定调用哪个已经开放的 Chub 能力，但不能绕过 Chub 自己构造命令、路径或设备操作；Chub 只接收结构化、范围固定的请求，不负责理解开放式自然语言。

整体关系如下：

```text
消息渠道 / 自然语言
            │
            ▼
     OpenClaw Gateway
     Agent、模型、记忆
            │
       受限工具调用
            │
            ▼
          Chub
固定 API、状态确认、安全边界
            │
            ▼
macOS / Ubuntu / Codex / Debug Chrome
```

### 7.2 OpenClaw 负责

- 自然语言入口和消息渠道。
- Agent、模型选择和上下文。
- 长期记忆和信息汇总。
- 将用户意图转换为受控工具请求。
- 跨设备的交互和任务通知。

### 7.3 Chub 负责

- 节点状态和本机服务维护。
- 固定、经过验证的设备操作。
- Codex PTY 会话生命周期。
- Debug Chrome 与浏览器自动化。
- 文件、日志和任务状态的受限读取。
- 异步操作的最终状态确认。
- 操作日志和本地安全边界。

Chub 不需要成为一个通用 Agent 框架，也不应为了接入 OpenClaw 开放任意 Shell、任意文件路径或完整浏览器权限。它可以继续把高价值设备能力封装成稳定、受认证、结果明确的 API，再由 OpenClaw 将这些 API 作为工具调用。

### 7.4 重叠能力的处理

| 重叠能力 | 责任方 | 处理方式 |
| --- | --- | --- |
| 用户对话与自然语言理解 | OpenClaw | Chub 不增加通用对话和 Agent 能力 |
| 节点设备登记 | 分层负责 | OpenClaw 管理 Agent 可见的逻辑节点；Chub 管理本机身份、平台和受控能力 |
| 节点状态 | Chub | Chub 采集真实状态，OpenClaw 只查询、汇总和解释 |
| 系统命令 | Chub | 不开放通用命令；只提供固定操作，OpenClaw 不直接执行 Chub 节点 Shell |
| 浏览器自动化 | Chub | 继续复用已验收的 Debug Chrome 和固定流程，OpenClaw 只触发任务并读取结果 |
| Codex 会话 | Chub | Chub 管理 PTY 生命周期；OpenClaw 不接管 Codex 内部权限、审批和终端协议 |
| 定时任务 | 按任务确定唯一调度方 | 同一任务只能由一方调度，避免重复执行；设备流程仍由 Chub Runner 完成 |
| 任务状态 | Chub | Chub 是执行状态的事实来源，OpenClaw 不根据请求返回自行推断成功 |
| 通知与结果摘要 | OpenClaw | 读取 Chub 最终结果后，通过消息渠道通知用户 |
| 操作日志 | Chub | Chub 记录真实设备操作；OpenClaw 保留对话和工具调用上下文，两者不混为一份审计日志 |
| 凭证 | 各自管理 | OpenClaw 不读取 Hub Token 之外的 Chub 本机秘密；Chub 不接收模型或频道凭证 |

### 7.5 事实来源

集成后需要明确每类状态由谁最终解释：

- Agent、会话、消息路由和记忆状态以 OpenClaw 为准。
- 节点系统状态、Chub 服务状态、自动化任务和 Codex PTY 状态以 Chub 为准。
- OpenClaw 发起调用后只能显示“已请求”或“执行中”，直到 Chub 返回可验证的最终结果。
- Chub 不记录或推断 OpenClaw 的用户身份，只记录固定来源、操作、目标和请求来源信息。
- 跨系统错误应保留原始语义，例如未认证、节点离线、任务繁忙、执行失败和结果超时不能统一显示为“Agent 执行失败”。

### 7.6 首期接口边界

首期集成只允许 OpenClaw 通过专用适配工具访问少量 Chub API：

```text
OpenClaw Agent
      │
      ▼
Chub 专用 Tool / Adapter
      │  固定字段、固定超时、Hub Token
      ▼
Chub 受控 API
```

接口约束：

- 不向 Agent 暴露 Hub Token，凭证由适配层保存和注入。
- 不接受任意 URL、主机、命令、工作目录或文件路径。
- 每个工具对应一个明确的 Chub 能力，不提供通用 API 转发器。
- 读取接口限制数据量并保持脱敏；操作接口沿用 Chub 认证、锁和操作日志。
- 长时间任务返回任务 ID，由适配层查询最终状态，不能用 HTTP 受理成功代替任务成功。
- Chub 不依赖 OpenClaw 才能运行；OpenClaw 不可用时，原有 Web 管理入口仍然有效。

### 7.7 当前不打通的能力

首期明确不做：

- 将 Chub 节点直接注册成拥有完整系统权限的 OpenClaw Node。
- 让 OpenClaw 直接连接 Chub 的 tmux、ttyd 或 Debug Chrome。
- 在 Chub 中实现 OpenClaw 的消息渠道、记忆或多 Agent 能力。
- 让两套系统共同调度同一个自动化任务。
- 同步两边的全部日志、配置、凭证和会话内容。
- 通过一个通用工具开放 Chub 全部 API。

## 8. 建议的分阶段验证

### 第一阶段：只读连接

让 OpenClaw 通过专用适配工具读取：

- Chub 节点状态。
- Chub 服务健康状态。
- 自动化任务运行结果。
- Codex 会话状态。
- 经过限制和脱敏的运行摘要。

这一阶段验证认证、数据格式、超时、错误反馈和手机端体验，不允许状态变更。建议先固定一个 Chub 节点，避免首轮同时引入多节点发现与路由。

### 第二阶段：低风险操作

在只读调用稳定后，选择少量可恢复、影响范围明确的操作，例如运行一个已经验收的自动化任务。每项操作继续由 Chub 完成输入校验、并发控制和最终状态确认。

OpenClaw 只负责发起请求、展示进度和返回结果，不能把“请求已接收”直接解释为成功。

### 第三阶段：确认与主动任务

根据真实需求评估：

- OpenClaw 定时生成设备与任务摘要。
- Chub 高风险操作通过 OpenClaw 请求人工确认。
- 任务完成或失败后由 OpenClaw 发送通知。
- 多个 Chub 节点是否需要统一发现和聚合。

只有在前两阶段证明有稳定价值后，才考虑更深的节点协议、双向事件或多节点管理。

## 9. 当前结论

OpenClaw 提供了 Chub 当前缺少的自然语言入口、消息渠道、Agent、模型、记忆和任务编排能力；Chub 则提供 OpenClaw 通用工具体系中缺少的本项目固定能力、严格边界和最终状态验证。

后续最值得验证的方向不是重做 OpenClaw，也不是把 Chub 直接改造成 OpenClaw Node，而是让 OpenClaw 成为上层智能入口，让 Chub 继续作为下层可信设备服务。首个验证场景应从只读状态汇总开始，再根据真实使用价值逐步开放低风险操作。
