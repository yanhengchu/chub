# Hub 第二阶段产品目标

> 当前状态：方向已确定，具体节点能力待结合实际使用场景持续探索。

## 1. 阶段定位

第一阶段完成了 macOS 与 Ubuntu 独立 Hub Node、受控 API、移动端管理页面和
开发目录服务化运行，验证了双平台节点能够安全、稳定地提供基础状态与白名单
任务。

第二阶段不急于建设统一多节点中心，而是优先扩展真正有用的节点能力。通过持续
使用、增加和调整能力，验证 Hub 能否降低个人设备的日常查看、检查和维护成本，
并为后续多节点统一入口积累真实需求。

## 2. 阶段目标

### 2.1 发现高价值使用场景

- 梳理 macOS 与 Ubuntu 日常管理中的重复操作和信息查询需求。
- 记录场景的使用频率、操作成本、风险和适用平台。
- 建立候选能力清单，但不要求一次确定完整范围。
- 优先选择能够在真实使用中持续验证价值的场景。

### 2.2 持续扩展节点能力

- 继续使用显式注册的白名单任务承载节点能力。
- 优先增加低风险、结果清晰的查询和检查能力。
- 按小批次交付，每批能力独立实现、测试和验收。
- 平台共性能力尽量复用，平台差异保持在对应平台实现中。
- 根据探索结果决定是否引入会改变设备状态的受控操作。

### 2.3 保持安全和可控

- Chub 不提供用户直接提交任意 Shell 命令的 API 或网页终端。
- 完整开发操作仅通过受管理的 Codex session 提供。
- 每项能力明确输入、输出、超时、失败语义和平台范围。
- 涉及状态变更的能力必须单独评估风险、确认交互和审计要求。
- 单项能力失败不得影响节点状态、其他任务和管理页面。
- 不因能力扩展降低现有 Token、日志脱敏和可信网络边界。

### 2.4 优化实际使用体验

- 保持手机端优先，新增能力在小屏幕上仍然清晰、易操作。
- 任务名称、说明和结果以用户可理解的方式展示。
- 根据能力数量变化改进任务查找、分类和结果呈现。
- 记录真实使用反馈，删除低价值能力，优化高频能力。

### 2.5 为后续阶段提供依据

第二阶段结束时，应能够回答：

- 哪些节点能力具有稳定使用价值？
- macOS 与 Ubuntu 的共性需求和平台特有需求分别是什么？
- 查询能力是否足够，是否需要受控的状态变更能力？
- 当前单节点页面是否仍能承载增加后的能力数量？
- 是否已经具备建设多节点统一入口的真实需求和能力基础？

### 2.6 探索手机端 Codex 开发代理

- 用户可从手机在用户目录、Workspace 和 Chub 三个固定入口中选择工作目录并
  创建 Codex session。
- 一个节点可以保存多个 session，并支持进入、恢复、停止、归档和删除。
- Session 支持需求讨论、固定分析任务和本地开发操作。
- 用户在 PTY 中直接使用 Codex CLI 原生的权限选择、Full access 和审批交互，
  Chub 不重复实现权限控制界面。
- Full access 包含修改文件、运行测试、Git 提交、推送及其他当前用户可执行的
  开发操作。
- Codex 可暂停任务等待用户补充信息或确认重要决策。
- 具体执行内核和交互协议在原型验证后确定。

## 3. 工作原则

- 真实需求优先，不为了丰富功能数量而增加能力。
- 小步交付，每次只增加少量相关能力。
- 查询优先，状态变更操作单独评审。
- 复用第一阶段的任务注册、执行、安全和页面结构。
- 每项能力都需要自动化测试和至少一个目标平台的实机验收。
- 具体能力清单允许根据使用反馈持续调整。

## 4. 阶段范围

第二阶段包含：

- 节点使用场景收集和候选能力管理。
- 公共、macOS 和 Ubuntu 节点能力扩展。
- 新增能力所需的受控 API 与页面展示调整。
- 任务结果可读性、分类和操作反馈优化。
- 对新增能力的自动化测试、实机验收和使用反馈记录。
- 可信 tailnet 内的手机端 Codex session 与 Full access 开发代理探索。

Codex 手机远程的候选方案、PTY 实验记录和待验证项见
[Codex 手机远程方案探索](CODEX_REMOTE_OPTIONS_PHASE_2.md)。

## 5. 安全与网络边界

- Tailnet 只包含用户自己的三台可信设备。
- Chub 必须监听节点自身的 Tailscale IP，不监听 `0.0.0.0`、localhost、普通
  局域网地址或公网地址。
- 保留第一阶段 Bearer Token 作为应用层认证。
- 不为 Full access 额外引入第二密码、TOTP 或 Passkey。
- 不启用 Tailscale Funnel，也不通过普通局域网代理 Chub。
- Session 必须支持取消当前执行、结束会话和清理 Codex 子进程。
- Codex PTY 入口仅在 Tailscale 监听条件成立时提供；Codex 内部权限模式由
  Codex CLI 自行管理。

Tailscale 默认 IPv4 地址来自 `100.64.0.0/10`，IPv6 地址来自
`fd7a:115c:a1e0::/48`。Chub 启动时检查配置的监听地址：非 Tailscale 地址只
影响现有基础功能的网络暴露提示，不阻止 Hub 基础服务启动，但必须将 Codex
PTY 标记为不可用。该限制必须在服务端生效，不得仅依赖页面隐藏入口。

地址范围依据 Tailscale 官方的
[IP 地址说明](https://tailscale.com/docs/concepts/ip-and-dns-addresses)与
[IPv6 说明](https://tailscale.com/docs/concepts/ipv6)。

## 6. 暂不纳入

- 多节点中心服务、自动发现、状态聚合和任务转发。
- 浏览器任意 Shell API、网页终端和脱离 Codex session 的命令执行入口。
- 定时任务、复杂工作流和无人确认的自动化操作。
- 多用户、复杂权限和公网直接部署。
- 独立发布目录、自动升级和版本回滚。
- 为尚未确定的能力预先设计插件系统或复杂扩展框架。

以上内容不代表永久排除，应在节点能力得到实际验证后重新评估。

## 7. 高层验收方向

第二阶段的最终验收标准将在首批能力确定后细化。高层要求包括：

- 新增能力来自明确的真实使用场景。
- 能力在适用平台稳定运行，失败和超时语义明确。
- 用户能够通过手机管理页面理解并使用新增能力。
- 新增能力不突破白名单、安全、日志和网络边界。
- 形成经过实际使用筛选的节点能力集合和下一阶段建议。
- 非 Tailscale 监听地址下，启动命令和应用日志均给出警告，Codex PTY
  保持不可用。
- Tailscale 监听地址下，可信设备可在通过 Chub Token 后创建 Codex session。
- 节点页移除低价值的“系统检查”和“版本信息”入口，暂时保留“Codex 检查”，
  并增加 Codex PTY Session 面板。
- 点击 Session 可进入独立终端子页面，返回后仍保持节点页原来的 PTY 面板状态。
- Session 列表统一展示本机全部未归档 Codex 会话，不区分创建入口；删除时同步
  永久删除 Codex 本地会话历史。
- 归档通过 Codex CLI 完成，归档后从 Chub 当前会话列表隐藏，但不删除历史。
