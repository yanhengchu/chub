# Hub 第二阶段产品目标

> 当前状态：Codex PTY 已完成 macOS、Ubuntu 和手机端验收，当前主线转向交付高价值节点维护能力。

## 1. 阶段定位

第二阶段不做多节点中心化平台，先继续扩展单节点的真实可用能力，重点验证个人设备在日常查看、检查和维护上的价值。

## 2. 阶段目标

- 找到高频、低风险、可持续验证的节点场景。
- 继续用白名单任务承载节点能力。
- 优先增加查询、诊断和可控的轻量操作。
- 保持手机端优先，结果和操作都要易读。
- 根据真实使用反馈删掉低价值能力。
- 探索手机端 Codex 会话管理入口，验证 PTY 是否足够承载远程开发。

## 3. 工作原则

- 真实需求优先，不为了“功能数”扩展。
- 小步交付，每次只加少量相关能力。
- 查询优先，状态变更单独评估。
- 复用第一阶段的任务注册、认证、日志和页面结构。
- 每项能力都要有自动化测试和实机验收。

## 4. 阶段范围

包含：

- 节点场景收集和能力管理。
- 公共、macOS 和 Ubuntu 能力扩展。
- 新增能力对应的 API 和页面调整。
- 结果展示、分类和交互反馈优化。
- 自动化测试、实机验收和使用反馈记录。
- 配置驱动的固定网站下载自动化基础能力和首页手动运行入口。
- 可信 tailnet 内的手机端 Codex 会话探索。

不包含：

- 多节点中心、自动发现和任务转发。
- 任意 Shell 入口、通用网页终端和脱离 Codex 会话的命令执行。
- 系统定时调度、跨任务复杂工作流和无人确认的状态变更自动化。
- 多用户、复杂权限和公网直连。
- 复杂插件系统和未验证的扩展框架。

## 5. 安全与网络边界

- Chub 必须监听节点自己的 Tailscale IP。
- 保留 Bearer Token 认证。
- 不启用 Tailscale Funnel。
- 不把 Hub 直接暴露到公网或普通局域网。
- Codex PTY 只在可信 tailnet 内启用。
- Codex 内部权限和审批仍由 Codex CLI 原生界面管理。

## 6. 高层验收方向

- 新增能力来自明确真实场景。
- 能力在适用平台稳定运行，失败语义明确。
- 手机端能理解并使用新增能力。
- 不突破白名单、安全和网络边界。
- 形成经过实际使用筛选的能力集合和下一阶段建议。

## 7. 页面信息架构

管理页面保持三个主要区域：

- 当前节点：展示节点身份和系统概况。
- Codex PTY：管理并接管本机 Codex CLI 会话。
- 节点维护：集中承载维护检查、服务操作和诊断信息。

顶层卡片统一使用“绿色分类说明 + 主标题”的标题结构，并按需提供简短辅助说明：

- `当前节点` + 节点名称。
- `远程开发` + `Codex PTY`。
- `节点维护` + `维护工具`。

新增能力应优先归入以上区域；只有形成独立、完整的操作流程时，才增加新的顶层卡片。

卡片交互保持相互独立：

- 每张动态卡片独立加载、刷新和展示错误，局部失败不影响其他卡片。
- 卡片上的“刷新”只更新本卡片；首次连接允许并行加载全部卡片。
- 卡片内操作默认只更新本卡片。服务重启等全局操作完成后，只定向刷新实际受影响的卡片。
- 动态卡片的右上角统一放本卡片“刷新”；“全部任务”“全部文档”等次级导航放在卡片底部右侧。
- 标题简介右侧最多放一个操作入口；桌面双列和手机单列都使用卡片自然内容高度。
