# Hub 第一阶段产品需求文档（PRD）

## 1. 文档目的

本文档用于明确 Hub 第一阶段需要交付的产品形态、范围和验收标准，并简要说明其所服务的整体产品方向。

其中，整体产品形态仅作为第一阶段的上位背景，后续可根据实际使用情况调整；第一阶段交付范围作为当前需求、设计、开发和验收的主要依据。

本文档不展开技术选型、工程目录、接口字段和具体实现方式。

## 2. 产品背景

用户拥有多台个人设备，例如 MacBook、Ubuntu 主机，以及未来可能接入的其他设备。这些设备运行环境和可用能力不同，目前缺少一个轻量、一致且安全的方式，让用户能够随时了解设备状态并完成常用操作。

Hub 希望降低管理个人设备的成本。用户不必登录每台设备或记忆具体命令，即可通过手机或电脑浏览器确认设备是否正常，并使用设备预先开放的能力。

## 3. 整体产品方向

### 3.1 方向性目标

Hub 的大致目标是成为一个面向个人设备的轻量管理入口。

每台设备可以接入 Hub，提供自身状态和经过授权的设备能力。用户可以查看设备、了解运行情况，并执行受控操作。随着使用场景增加，Hub 可逐步提供更统一的多设备管理体验。

整体形态大致包含：

- 运行在各设备上的节点能力。
- 面向用户的查看与操作入口。
- 对设备状态和可用能力的统一表达。
- 安全、可控的设备操作机制。

具体是否采用中心节点、独立控制台、节点互联或其他组织方式，不在当前阶段固定，将根据后续实际需求调整。

### 3.2 核心产品价值

- 用户能够快速确认个人设备是否在线、是否正常。
- 用户无需了解底层命令即可完成常用检查和受控操作。
- 不同设备以相对一致的方式展示状态和能力。
- 单台设备出现故障时，不应不必要地影响其他设备。
- 新设备和新能力可以逐步接入，而无需推翻已有使用方式。

### 3.3 核心概念

- **Hub**：面向个人设备管理的整体产品。
- **Hub Node**：运行在具体设备上的节点服务，负责提供本机状态和能力。
- **节点能力**：设备允许用户调用的查询、检查或操作能力。
- **任务**：第一阶段承载节点能力的主要形式，由用户主动触发并返回结果。

未来如果出现统一的多节点入口，可再根据实际方案定义 Hub Console 等新概念，当前不提前固定。

### 3.4 长期产品原则

- **安全可控**：设备能力必须有明确边界，用户输入不能直接变成不受约束的系统命令。
- **节点自治**：节点应尽可能独立工作，避免无必要的中心依赖。
- **跨平台一致**：不同平台保持相近的产品使用方式，同时允许提供平台专属能力。
- **渐进演进**：先解决明确问题，再根据真实使用需求增加多节点、自动化或智能化能力。
- **简单可维护**：优先满足个人场景中的可靠性和可理解性，避免过早引入复杂体系。

## 4. 第一阶段目标

第一阶段只验证独立 Hub Node 的产品价值和跨平台可行性。

目标是在 MacBook 和 Ubuntu 上分别运行一个 Hub Node，让用户可以通过手机或电脑浏览器独立访问每个节点，查看本机状态、执行白名单检查任务并查看 Hub 运行日志。

第一阶段完成后，应能够回答以下问题：

- 同一套 Hub Node 能否在不同平台稳定运行？
- 用户能否通过浏览器方便地了解设备状态？
- 白名单任务能否安全地替代一部分手工命令检查？
- 独立节点形态是否值得继续扩展为多设备管理产品？

## 5. 第一阶段交付形态

### 5.1 产品组成

第一阶段交付两个彼此独立的 Hub Node：

```text
手机或电脑浏览器
    │
    ├── MacBook Hub Node
    │   ├── 节点状态
    │   ├── 公共检查任务
    │   ├── Codex 检查
    │   └── Hub 日志
    │
    └── Ubuntu Hub Node
        ├── 节点状态
        ├── 公共检查任务
        ├── Docker 检查
        └── Hub 日志
```

两个节点使用相同的产品结构，但展示各自的设备信息和可用能力。用户分别访问两个节点，不提供统一的多节点首页。

Hub Node 除提供浏览器管理页面外，还应提供与页面能力对应的受控服务接口，用于健康检查、状态读取、任务查询与执行以及日志读取。具体接口形式在技术方案中定义。

### 5.2 用户访问方式

- 用户通过浏览器打开目标节点的独立地址。
- 节点运行在局域网或虚拟组网等可信网络中。
- 用户需要提供该节点认可的访问凭证，才能查看状态、执行任务和读取日志。
- 用户首次访问时可以录入访问凭证，并可选择保存在当前浏览器中。
- 凭证无效时，页面应明确提示，并允许用户重新录入或清除凭证。
- 不同节点的访问凭证分别管理，不默认共享。
- 健康检查可以在不提供凭证的情况下使用，但不得暴露敏感设备信息。
- 用户需要自行选择并访问目标节点，第一阶段不负责自动发现或汇总节点。

### 5.3 页面形态

每个节点提供一个轻量、适配手机浏览器的管理页面。页面至少包含：

- 节点名称、类型和 Hub Node 运行状态。
- CPU、内存、磁盘和运行时间等系统概况。
- 当前节点可用的任务。
- 最近一次任务的执行结果。
- Hub 近期运行日志的查看入口。

页面以完成操作为主，不要求复杂视觉设计、桌面级交互或独立前端工程。

### 5.4 节点状态能力

用户可以查看：

- 节点身份和设备名称。
- 当前操作系统及基础环境信息。
- CPU、内存和磁盘使用情况。
- 系统启动时间或运行时长。
- Hub 自身版本和运行状态。

状态信息应便于用户快速判断设备是否正常，不要求提供完整的系统监控能力。

### 5.5 任务能力

第一阶段采用白名单任务形式提供节点能力。

公共任务包括：

- 查看 Hub、Python 和节点的版本信息。
- 检查基础系统状态。

平台专属任务包括：

- MacBook：检查 Codex 是否可用及其版本信息。
- Ubuntu：检查 Docker 是否运行及其基础状态。

任务应满足：

- 用户只能执行当前节点明确提供的任务。
- 不适用于当前平台的任务不向用户展示。
- 每次执行均返回成功或失败，以及用户可理解的结果说明。
- 单次任务失败不能导致 Hub Node 停止服务。
- 任务应在合理时间内结束，避免长期占用节点。
- 第一阶段任务以查询和检查为主，不修改系统状态。

### 5.6 日志能力

- 用户可以查看 Hub 自身近期运行日志。
- 日志用于了解请求、任务执行和异常情况。
- 用户不能指定或读取设备上的任意文件。
- 日志不得展示访问凭证等敏感信息。
- 第一阶段不提供完整日志检索、归档或分析能力。

### 5.7 节点独立性

- 每个节点拥有自己的状态、任务、日志、配置和页面。
- 节点不依赖其他 Hub Node 才能启动或工作。
- 一台设备关机或离线，不影响其他在线节点正常使用。
- 第一阶段不进行节点间通信或任务转发。

## 6. 第一阶段用户场景

### 6.1 确认 Hub Node 是否可用

用户通过手机打开目标节点地址。页面能够正常加载时，用户可以确认目标设备上的 Hub Node 正在运行；节点离线时，第一阶段不通过其他节点主动展示其离线状态。

### 6.2 查看设备状态

用户打开节点页面，通过系统概况判断设备是否存在明显资源异常。

### 6.3 检查 MacBook 上的 Codex

用户在 MacBook 节点选择 Codex 检查任务，查看 Codex 是否可用、安装位置和版本信息，无需登录设备执行命令。

### 6.4 检查 Ubuntu 上的 Docker

用户在 Ubuntu 节点选择 Docker 检查任务，查看 Docker 是否运行、版本和容器概况，不对容器执行启动、停止或重启操作。

### 6.5 排查 Hub 运行问题

当任务失败或页面状态异常时，用户查看 Hub 近期日志，获取基本排查信息。

### 6.6 单节点离线

当 MacBook 或 Ubuntu 中的一台离线时，用户仍可正常访问另一台在线设备的 Hub Node。

## 7. 第一阶段范围边界

### 7.1 本阶段包含

- MacBook 和 Ubuntu 的独立 Hub Node。
- 节点健康检查。
- 节点和系统状态查看。
- 白名单任务列表与执行结果。
- 公共检查任务。
- Codex 和 Docker 平台检查任务。
- Hub 自身近期日志查看。
- 适配手机浏览器的简单页面。
- 基础访问保护。

### 7.2 本阶段不包含

- 多节点统一页面、状态聚合或自动发现。
- 节点之间的任务转发、选举或协作。
- 通用远程终端和任意系统命令执行。
- 远程桌面、窗口或设备 UI 控制。
- Docker 容器启动、停止、重启等管理操作。
- 定时任务、长任务、任务队列和任务历史。
- 用户系统和复杂权限管理。
- AI 自然语言解析或自动修改代码。
- 数据库和复杂前端工程。
- Windows 节点。

除违反长期安全原则的能力外，以上内容仅为第一阶段非目标，不代表永久排除。远程控制、自动化等能力可在后续根据实际需求、安全边界和维护成本单独评估；不受约束的任意命令执行不属于产品能力。

## 8. 安全与使用约束

- Hub Node 默认仅部署在用户控制的可信网络中，不作为公网服务开放。
- 除最小健康检查外，节点信息和操作需要访问凭证。
- 节点只提供预先定义的白名单任务。
- 用户不得提交任意命令、脚本或可执行文件。
- 用户提供的任务参数必须受到明确约束和校验。
- Hub 不记录或返回访问凭证等敏感信息。
- 日志查看只覆盖 Hub 自身日志。
- 第一阶段任务只进行状态查询和环境检查，不主动修改设备状态。

## 9. 第一阶段验收标准

### 9.1 跨平台与独立运行

- 同一版本、同一套 Hub Node 工程能够分别运行在 macOS 和 Ubuntu 上，平台差异通过节点配置或平台能力适配处理，不维护两套独立产品实现。
- 两个节点均可通过可信网络独立访问。
- 任一节点离线时，另一节点仍可正常使用。
- Hub Node 的基础能力通过受控服务接口提供，可以被浏览器页面使用，并为后续其他客户端复用。

### 9.2 浏览器使用体验

- 用户可以通过手机浏览器完成主要操作。
- 用户能够明确识别当前访问的是哪一台设备。
- 首页无需额外操作即可展示节点身份、Hub Node 运行状态以及 CPU、内存、磁盘等系统概况。
- 页面能够清楚展示任务执行中、成功或失败状态。
- 用户可以录入、保存、更换和清除当前节点的访问凭证，凭证无效时能够看到明确提示。

### 9.3 节点状态

- 用户可以查看节点身份、平台和主要系统资源状态。
- 健康检查不泄露敏感节点信息。
- 未授权用户不能查看完整节点状态。

### 9.4 任务执行

- 两个平台都可以执行公共检查任务。
- MacBook 节点可以检查 Codex。
- Ubuntu 节点可以检查 Docker。
- 用户无法执行未注册或不属于当前平台的任务。
- 用户无法通过任务参数执行任意系统命令。
- 任务失败时至少展示失败任务和原因说明，并且不影响节点继续运行。
- 任务超过允许的最长执行时间后能够结束，并返回明确的超时结果。

### 9.5 日志与安全

- 用户可以查看 Hub 自身近期日志。
- 用户无法通过日志功能读取任意文件。
- 访问凭证不得在页面中明文回显，也不得出现在任务结果或日志中。

## 10. 第一阶段验证重点

第一阶段除验证功能是否完成外，还需要观察：

- 手机访问是否足够方便，是否能替代部分 SSH 检查。
- 节点状态是否提供了足够但不过量的信息。
- Codex 和 Docker 检查是否是有实际价值的首批任务。
- 白名单任务模型是否便于继续增加设备能力。
- 分别访问节点是否能够满足当前使用，何时真正需要统一入口。

这些结果将作为后续调整整体产品形态和规划下一阶段的依据。

## 11. 风险与限制

- 节点可用性受到设备电源、网络和虚拟组网状态影响。
- 第一阶段面向个人可信网络，不能替代完整的公网安全体系。
- 不同操作系统和软件版本可能导致检查结果存在差异。
- 第一阶段只适合短时查询和检查任务，不适合长时间运行的操作。
- 节点数量增加后，分别访问地址的体验会下降，但当前不提前确定解决方案。

## 12. 文档维护原则

出现以下变化时，应同步更新本文档：

- 整体产品方向发生明显变化。
- 第一阶段新增或删除核心交付能力。
- 安全、节点独立等核心原则发生变化。
- 第一阶段验收标准发生变化。

技术方案、工程结构、接口设计和开发任务应在后续设计文档或任务文档中维护，避免将 PRD 演变为实现说明。
