# AGENTS.md

## 项目定位

Chub 是个人维护的轻量设备管理服务，主要技术栈为 Python 3.12、FastAPI、Pydantic、Jinja2、原生 JavaScript/CSS 和 Shell。支持 macOS LaunchAgent 与 Ubuntu systemd user service。

项目维护者主要负责提出需求、讨论方案和验收实际运行结果；编码 Agent 负责在后台阅读与修改代码、完成必要文档同步和基本验证。协作以功能是否符合预期为主，不要求维护者参与代码实现细节。

## 协作方式

- 用户描述问题或需求后，优先直接分析和解决，不输出冗长计划、完整思考过程或大段技术推演。
- 中间沟通保持简短，只在需要确认方向、存在风险或执行时间较长时说明关键进展。
- 最终回复只保留核心内容和明确结论，让用户能够快速判断功能是否符合预期。
- 功能开发完成后不要向用户粘贴实现代码，也不要默认要求用户 review 代码业务逻辑。
- 用户验收以实际页面、交互、运行状态和最终效果为主；应主动完成基本运行验证，并说明用户需要重点体验什么。
- 只有当实现选择会明显影响产品行为、风险或后续维护时，才简要解释必要的技术原因。
- 需求只有小范围空白时，可以根据现有实现做合理假设，并在交付结果中说明。
- 改动尽量小步、聚焦，避免顺手进行无关重构。
- 优先复用现有目录结构、代码风格、响应格式和页面交互。
- 发现更大的结构问题时，先简要说明，但不要因此偏离当前任务。
- 用户仅要求分析、评审或解释时，不要擅自修改代码。
- 修改功能后应检查相关测试、文档和跨平台影响。

## 项目结构

- `app/api/`：FastAPI 接口。
- `app/core/`：配置、安全、日志、平台检测等基础能力。
- `app/services/`：可复用服务逻辑。
- `app/tasks/`：白名单维护任务及平台实现。
- `app/codex/`：Codex PTY 会话、票据和连接管理。
- `app/web/templates/`：Jinja2 页面模板。
- `app/web/static/`：原生 JavaScript 和 CSS。
- `scripts/`：安装、服务管理和辅助脚本。
- `tests/`：pytest 测试。
- `config/`：跨平台配置示例；本机配置不提交。

## 实现约束

- API 保持统一的 `ApiResponse`/`ApiError` 响应结构。
- 受保护接口必须继续使用 Hub Token 认证；健康检查除外。
- 不允许客户端提供任意文件路径或任意系统命令，只能使用后端固定映射或白名单。
- 不在日志、响应、测试输出或文档中暴露 Token、Authorization、终端票据等敏感数据。
- 页面渲染外部内容和日志时使用 `textContent`，不要使用 `innerHTML`。
- 文件读取必须限制行数、字节数和单行长度，避免一次加载无界内容。
- 后台服务行为需同时考虑 macOS 和 Ubuntu；无法实机验证的平台要在结果中说明。
- Web 自重启使用 `scripts/chub-web-restart`，不要从服务内部执行 `bootout` 后再依赖同一子进程启动服务。
- 异步操作不能仅凭子进程成功创建就宣告完成；重启等操作应通过实例 ID、健康状态或其他最终状态确认结果。
- 新增维护操作时，同时考虑操作日志的 `requested`、`started`、`succeeded` 和 `failed` 状态；无法由旧进程确认最终结果时，不要伪造成功日志。
- 不随意引入前端框架、数据库、任务队列或其他复杂基础设施。

## 日志规范

- 操作日志用于记录用户触发的维护行为；运行日志用于排障，两者不要混为单一语义。
- 操作日志至少包含时间、操作 ID、动作、状态、目标和来源 IP，不宣称具备用户身份审计能力。
- 操作成功和失败都应记录；日志记录本身不得改变原操作的响应语义。
- 日志来源必须由后端固定枚举映射，禁止客户端传入任意路径。
- 页面读取必须限制行数、读取字节数和单行长度；分页遇到超长单行时仍必须保证游标向前推进。
- 页面展示可以截断超长行；日志下载应保留完整内容并进行敏感信息脱敏，同时设置合理的文件大小上限。
- 日志文件及轮转文件默认使用 `600` 权限，不依赖系统默认 umask 提供安全性。
- 日志轮转、文件增长和服务重启期间允许安全降级，但不能导致无限分页、任意文件读取或服务不可用。

## 配置与本机文件

- `.env` 和 `config/settings.local.yaml` 只保存本机配置，不应提交。
- 不覆盖用户现有配置；新增配置项应提供安全、向后兼容的默认值。
- 不把真实 Token、IP、主机名或用户目录写入示例配置和测试夹具。
- 服务直接依赖当前工作区和 `.venv`，涉及服务定义变化时说明是否需要重新执行安装。

## 验证

根据改动范围选择最小充分验证；交付前通常至少运行相关测试。

```bash
.venv/bin/python -m pytest tests/test_relevant_file.py
```

涉及公共配置、API、中间件或跨模块改动时运行全量测试：

```bash
.venv/bin/python -m pytest
```

Shell 脚本至少进行语法检查：

```bash
bash -n scripts/<script-name>
```

提交 review 前检查补丁格式和工作区：

```bash
git diff --check
git status --short
```

Web 交互或布局变化应尽量进行实际浏览器验证。不要为了验证重启逻辑而意外中断用户正在使用的服务；任务明确授权重启时除外。

涉及异步服务操作时，验证最终状态而不只是 HTTP 200，例如同时确认：

- 服务 PID 或实例 ID 已变化。
- 健康检查恢复。
- 预期操作日志已经写入。
- 失败路径不会被前端误报为成功。

涉及文件、日志或分页时，至少考虑以下边界：

- 文件不存在、为空或读取失败。
- 单行超过读取窗口。
- 非 UTF-8 内容。
- 敏感信息跨越截断边界。
- 非法来源、路径和重复查询参数。
- 文件权限、大小上限与轮转。

测试必须使用临时目录和测试配置，不向项目真实日志、本机配置或用户目录写入测试数据。测试替身不能执行真实的 `launchctl`、`systemctl`、删除或重启操作。

## Review 清单

完成实现后，从维护者视角再检查一次：

- 页面提示的“成功”是否代表最终状态，而非仅代表请求已接收。
- API 是否保持认证、统一响应和安全响应头。
- 是否存在任意命令、任意路径、未限制文件读取或敏感信息泄漏。
- 新配置是否向后兼容，是否错误覆盖本机配置。
- macOS 与 Ubuntu 的服务管理差异是否已处理或明确列为限制。
- 新增文件是否应纳入 Git，生成文件、缓存和本机数据是否被排除。
- 是否留下无用兼容接口、重复逻辑或与当前任务无关的改动。

## Git 与提交

- 默认不创建分支、不提交、不推送，也不创建 PR。
- 默认先让维护者验收本地运行效果，再决定是否提交；不要求维护者逐文件 review 代码。
- 不处理、回退或提交与当前任务无关的工作区改动。
- 如果之后被要求提交，只包含当前任务相关文件，并在提交前确认 diff。
- 禁止使用 `git reset --hard`、`git checkout --` 等破坏性命令清理用户改动。

## 文档

- 不要求每个小改动都先修改文档。
- 功能说明、结构、配置或任务状态明显变化时，同步必要文档。
- 优先维护 `README.md`、架构说明和任务清单；避免为简单实现新增重复文档。
- 文档中的命令、路径和接口必须与当前代码一致。

## 交付说明

最终回复保持简短，通常只包含：

- 功能是否已经完成，以及实际效果。
- 已执行的关键验证和验证结论。
- 用户需要重点验收的页面、交互或运行行为。
- 仍然存在的风险、限制或未验证平台。
- 是否需要重新安装、重启或执行额外操作才能生效。

除非用户明确要求，不要在最终回复中粘贴代码、展开实现过程、罗列大量修改文件或详细解释内部业务逻辑。
